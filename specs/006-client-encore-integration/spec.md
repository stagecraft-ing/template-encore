# 006 — Client Encore integration: SPA auth stores and the committed typed client

## 1. Purpose

The two Vue SPAs (`apps/web`, `apps/web-internal`) consume the Encore
backend's response payloads natively. Both Pinia auth stores read
Encore-shaped responses directly, obtain the CSRF token from the dedicated
endpoint body, and target the Encore development server via the Vite proxy.
A generated typed client committed at `apps/web/src/lib/encore-client.ts`
provides a drift-proof reference for the typed endpoint shapes.

## 2. Territory

This spec owns the two Pinia auth stores and the committed typed client.
The Encore API surface those stores call is owned by spec 003 (auth
service) and spec 004 (BFF gateway). The static serving wiring (Vite
`build.outDir`) is owned by spec 005.

## 3. Behavior

### 3.1 Encore-native response payloads

#### FR-001 — No envelope unwrapping

The auth stores MUST consume Encore-native response payloads directly. No
`{ success, data }` / `{ error: { message } }` envelope unwrapping is
present.

| Endpoint | Payload shape consumed |
|----------|----------------------|
| `GET /api/v1/auth/me` | bare `MeResponse` (`{ id, email, name, roles, ssoProvider, isActive, lastLoginAt, createdAt }`) |
| `GET /api/v1/auth/status` | bare `{ authenticated, drivers }` |
| `POST /api/v1/auth/logout` | `{ redirectUrl }` at top level |
| `GET /api/v1/auth/{driver}/login` | redirect; no body parsing |
| Error responses | top-level `{ code, message, details }` (Encore error shape) |

### 3.2 CSRF contract

#### FR-002 — Token fetch and replay

The auth stores MUST implement the double-submit CSRF contract against the
Encore backend (spec 002 INV-4):

1. The CSRF token is obtained by calling `GET /api/v1/auth/csrf-token`,
   which returns `{ token }` in the response body and sets the httpOnly
   CSRF cookie.
2. The token value is held in memory and replayed as the `X-CSRF-Token`
   request header on every state-changing call (login, logout, any
   mutation).
3. The in-memory token MUST be cleared in the `finally` block of `logout()`
   so a stale value is not replayed after a subsequent re-login.
4. `ensureCsrfToken()` MUST deduplicate concurrent in-flight fetches (reuse
   the in-flight promise) to avoid multiple simultaneous `GET csrf-token`
   calls when concurrent mutations are issued.

The httpOnly session-cookie model (spec 002 INV-3) is unchanged; this spec
governs only how the client *reads* responses and *obtains* the CSRF token.

### 3.3 Security hardening in the auth stores

#### FR-003 — SLO redirect restriction

`POST /api/v1/auth/logout` returns a `redirectUrl`. Before assigning it to
`window.location.href`, the store MUST validate the URL via
`new URL(...).protocol` and reject any scheme that is not `http:` or
`https:` (defence-in-depth against `javascript:` and `data:` injection,
even though the URL is backend-supplied).

### 3.4 Vite dev proxy

#### FR-004 — Proxy target

`apps/web/vite.config.ts` and `apps/web-internal/vite.config.ts` MUST
configure the `/api` dev proxy to target the Encore development server at
`http://localhost:4000`. The `/api/v1` URL prefix is unchanged; no path
rewrite is needed.

### 3.5 Committed typed client

#### FR-005 — Generated artifact, committed

A typed HTTP client generated by `encore gen client` MUST be committed at
`apps/web/src/lib/encore-client.ts`. Its purpose is:

- A fresh-clone build succeeds without first running the Encore CLI.
- It is the drift-proof reference for typed endpoint shapes (`me`,
  `drivers`, health/info).

The file carries a `DO NOT EDIT` banner. It is regenerated via
`npm --prefix apps/api run gen:client`. It is **not** coupling-claimed by
any spec (like a lockfile, claiming it would couple every backend API
change to this spec); its freshness is enforced by the CI staleness gate
in spec 007.

`apps/web/tsconfig.json` targets ES2022 or later (the generated client
emits `String.replaceAll`).

#### FR-006 — Auth-store isolation from the generated client

The auth stores MUST use the existing axios layer for all cookie, CSRF, and
raw-endpoint calls. Calling the generated client's mutating methods
directly would bypass the INV-4 double-submit contract (the generated
client issues raw fetches with no CSRF replay). Mutating calls MUST go
through the auth store (or a future CSRF-aware wrapper), never the
generated client directly.

## 4. Acceptance criteria

**AC-1.** Both SPAs typecheck (`npm run typecheck`) and build
(`npm run build:web`, `npm run build:web-internal`) against the Encore-
native payload shapes and the committed typed client.

**AC-2.** A smoke test against a running `encore run` confirms:
- `GET /api/v1/auth/me` returns a bare profile object (no `{ success, data
  }` wrapper).
- `GET /api/v1/auth/csrf-token` returns `{ token }` and sets the httpOnly
  cookie.
- `POST /api/v1/auth/logout` succeeds with the `X-CSRF-Token` header
  present.

**AC-3.** `npx spec-spine compile` exits 0; `npx spec-spine lint
--fail-on-warn` exits 0; `npx spec-spine couple --base origin/main` exits
0 for the two `auth.store.ts` files and the typed client owned here.

**AC-4.** The unit test suite for the auth stores (`npm test`) is green;
CSRF dedup, logout token-clear, and SLO redirect validation are covered.

## 5. Out of scope

- **Serving the SPA from Encore** (`api.static`, `build.outDir` wiring):
  owned by spec 005.
- **Typed client for `apps/web-internal`**: the `apps/web-internal` store
  is owned here; a typed client for that app is a future addition if
  needed.
- **Wiring the typed client into the auth stores**: the stores continue
  to use the axios layer. Using the generated client for auth calls would
  require a CSRF-aware wrapper; that is a deliberate future enhancement,
  most valuable once typed domain endpoints exist.
- **CI client-staleness gate**: the regeneration and diff-check automation
  is owned by spec 007.
